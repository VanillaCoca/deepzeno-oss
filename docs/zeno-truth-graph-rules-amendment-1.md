## 0. 本修订案不改变什么

v1 的以下决策全部保留，本案与之兼容并强化：

- **默认不画关系线，选中后只点亮单链**（v1 §1.1）—— 本案通过"位置承担结构"使其更成立，而非推翻。
- 两层视图、同屏并置（v1 §1、§2）。
- 绿红唯一语义、四形状、线型语言、色盲冗余信号（v1 §4）。
- truth 只读、一切判断变更回 sandbox（v1 §7）。
- 全局前提折叠条（v1 §6.3）、superseded 默认藏（v1 §8）—— 这两条 v1 已规定但实现未跟上，本案升级为**必须实现**。

## 1. 诊断（为什么要修订）

当前实现的全貌层用 rectpacking 排布，**位置不携带任何含义**，全部语义压在颜色上 —— 而颜色是较弱的分类通道，位置才是最强通道。这是"便利贴感"的根因一。根因二是缺叙事主轴：节点全是"答案"，它们所回答的问题在视图上不可见。

修复方向**不是画线**（Obsidian 全局图 hairball 是反面教材，v1 不画线的决策正确），而是让位置承担结构，从而根本不那么需要线。

## 2. 语义分层：Topic 内位置 = 含义

Topic 内部废弃 rectpacking，按五条横向带自上而下排布。读左侧带标签即读完这个 topic 的叙事：

| 带 | 内容 | 回答的问题 |
|---|---|---|
| **锚** | 该 topic 的锚节点（见 §5） | 这个 topic 为什么存在 |
| **前提** | global assumptions 折叠条（v1 §6.3） | 基于什么假设 |
| **已定** | status=active 的 truth | 确认过什么 |
| **待决** | open_question + pending 候选 | 现在卡在哪 |
| **已排除** | rejection / superseded，默认折叠 | 排除过什么 |

- **待决带是唯一视觉焦点**（琥珀）；已定沉为地基；已排除收进抽屉。回访用户最需要的问题是"现在卡在哪"，视图必须直接回答。
- 候选方案挂在它所回答的 open_question 之下（缩进 + 短角标连线）。这是 IBIS 结构：一切沉淀挂在其回答的问题下面，未经训练的用户也能直接读懂。
- 带内布局仍由 ELK 做稳定增量排布，空间记忆规则（v1 §1.1）不变。

## 3. 自我说明的节点（落实 v1 §4.1 / §4.7 的冗余编码）

每个节点 = **类型图标 + 一词类型章 + 标题**：

- `✓ 已确认`（绿）/ `? 待决问题`（琥珀）/ `候选 · 待确认`（紫，虚线描边 —— 虚线=未定，沿用 v1 §4.3）/ `✗ 已排除`（红 + 删除线 + "被 X 取代 →"）/ `前提`（灰，锚形图标）。
- 目的：初次使用者**不需要图例**，卡片自我说明；红绿色盲天然覆盖；颜色从唯一编码降级为强化信号。这是对最高信条（看一眼就懂）的直接落实。

## 4. 分层密度：版面按"当前决策相关性"分配，不按节点数量均分

| 生命周期 | 形态 | 为什么 |
|---|---|---|
| 待决（需要用户行动） | 完整卡片 | 卡片是"请注意我"的形态 |
| 已定 | 单行列表行（✓ + 标题 + 确认日期），点击进单链 | 已定只需"在场可查" |
| 前提 | 一行折叠条「基于 N 条前提」 | v1 §6.3 原文 |
| 已排除 | 一行折叠计数「已排除 N 项」，展开见全文 + 取代关系 | 抽屉而非删除 |

- **恒定可见区推论**：项目变老，增长的绝大多数是"已定"紧凑行，打开时可见区域大小近似恒定。这是本案对"拥挤"问题的根本解。
- 已排除项的残余价值：回顾"当时为什么不这么干" + 防重复提案 —— **提取层再提议任何方案前必须查已排除项**（Lixian 规则）。
- 若待决带拥挤（同 topic 悬而未决 > ~5 项）：**不做布局妥协**，触发产品提示"待决过多，建议收敛"。拥挤是产品信号，不是排版问题。

## 5. 锚的多态：topic 因何存在

### 5.1 本体论修正

**目标不是思考的原点，问题才是；目标是"被采纳的答案"，是探索的产物。** 典型演化：担忧/好奇（沙盒）→ 问题成形（建 topic）→ 探索沉淀候选方向 → 某候选被确认 → 固化为目标。

### 5.2 三种锚

| 锚 | topic 类型 | 例 |
|---|---|---|
| 想搞清楚的问题（open_question） | 探索型 | 「极端情形下人民币大幅贬值，资产如何避险」 |
| 已采纳的目标（goal, status=active） | 执行型 | 「3 年内在定居城市拥有自住房」 |
| 持续关注的领域（原则/标准集，无终点） | 维护型 | 「生活与家庭」 |

维护型锚对应 PARA 意义上的 Area：没有完成态，沉淀的是原则与标准。锚位显示领域名 + 该领域的活跃原则数。

### 5.3 创建门槛与转化规则

- **topic 的创建门槛不是"有目标"，而是"形成值得沉淀的引力中心"。** 操作化（Lixian）：当一批相互关联的新节点塞不进任何现有 topic 时，以串起它们的那个问题为锚建 topic。连问题都提不出来的节点留在沙盒（topicId=null）。**不存在无锚的 topic。**
- 转化：探索型 topic 中某候选被确认为 goal 后两种出口 ——
  1. goal 接管本 topic 锚位，原锚问题被 `resolves`；
  2. 体量足够大时分裂为新执行型 topic，写 `topic_relations` 记 lineage。
- **goal 本身是有完整生命周期的 IR node**：status=pending 即"候选目标"，出现在待决带，与其他候选无异，走同一套确认制。

### 5.4 目标可追溯（本案核心产品价值）

- 执行型 topic 的锚位必须显示出处：「由 X 的探讨产出 →」，点击进入跨 topic 单链。用户（和用户的未来自己）随时能回答"这个目标是怎么来的"。
- **project 层没有目标字段。** project 的目标 = 各 topic 活跃 goal 的集合（涌现）；目标变化 = 旧 goal 被 supersede —— 与 v1 §7 truth 只读铁律完全自洽。
- 定位声明：在一般工具里目标是公理，在 Zeno 里**目标是有出处的结论**。

## 6. 多 Topic 全貌布局

- **每个 Topic 一条脊柱**（§2 五条带），Topic 容器之间仍 rectpacking 稳定排布（v1 §1.1 不变）。
- 全貌顶部增加**全局待决聚合条**：「全项目 N 项待决」，横向列各 topic 待决问题摘要，点击 → 平移聚焦到对应 topic（不换页，遵守 v1 §2 同屏原则）。
- 为什么不用"全项目共享五条带"：锚是 topic 级概念，共享带会把锚与其内容拆散、叙事断裂，且新节点引发跨带全局重排，破坏空间记忆。

## 7. 数据层影响：无

`kind` / `status` / `resolves` / `supersededBy` / `topic_relations` / `topicId=null` 已足够支撑全部规则。落地位置：

- **提取层（Lixian）**：topic 创建与锚指定规则（§5.3）、goal 固化判定、再提议前查已排除项（§4）。
- **渲染层（Codex）**：五条带布局（§2）、类型章（§3）、密度分档与折叠（§4）、全局待决聚合条（§6）、锚位出处链接（§5.4）。

## 附 A：本修订案新增决策清单

1. Topic 内位置=含义（五条带），废弃 topic 内 rectpacking
2. 节点自我说明：图标 + 类型章 + 标题；颜色降为强化信号
3. 密度跟生命周期：完整卡片只给待决；已定单行；前提/已排除折叠
4. 已排除 = 折叠计数可展开，抽屉而非删除；提取层再提议前必须查
5. 待决拥挤触发产品提示，不做布局妥协
6. 锚三态：问题（探索型）/ 目标（执行型）/ 领域（维护型）；无锚不成 topic
7. topic 创建门槛 = 引力中心，非目标
8. goal 是有生命周期的节点；候选目标进待决带走确认制
9. 执行型锚必须显示出处；目标跨 topic 可追溯
10. project 无目标字段，目标由各 topic 活跃 goal 涌现
11. 全貌 = 每 Topic 一条脊柱 + 全局待决聚合条
